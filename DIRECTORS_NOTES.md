## Current Canon

### Project thesis
Teach a small local language model to ground musician-native semantics (phase locking, groove, brittleness, interference, etc.) in a bounded synthetic world where those semantics are computable from state. Evaluate on **intervention grounding** — does the model's suggested action actually move the sim the way it claimed?

### Toy world — Coupled Oscillator Garden (v0)
- `N = 8` nodes in the unit square.
- Kuramoto-style phase coupling with distance kernel `Kᵢⱼ = K₀ · exp(−dᵢⱼ/σ)`.
- Optional white-noise perturbation `ξᵢ(t)` with amplitude `η`.
- Per-node fields: `pos, ω₀, θ, A, γ, voice`.
- Audio is **pulse-driven**: each phase-zero crossing fires a percussive grain with a fixed-pitch voice; stereo pan from `pos.x`. Audio-rate tone coupling is deferred to v0.1.
- Scheduled (not real-time) events: `nudge`, `impulse`, `move`, `setK`.

### Run exports
`audio.wav` (22050 Hz stereo), `state.npz` (`θ`, `A` at 200 Hz), `events.jsonl`, `topology.json`, `labels.json`.

### Ontology (13 labels, multi-label, detector-grounded)
`phase_locked`, `drifting`, `polyrhythmic`, `phase_beating`, `dominant_cluster`, `brittle_lock`, `unstable_bridge`, `groove`, `groove_collapse`, `beat_bloom`, `tension_break`, `flam`, `regime_change`. Each has a deterministic detector; labels are ground truth, not model outputs. Detector confidence is **evidence margin** (how strongly / stably the defining condition was met), not epistemic uncertainty.

### Data pipeline
Regime-template sampler (`LOCKED`, `DRIFTING`, `TWO_CLUSTER`, `BRITTLE`, `SWEEP`, `PERTURBED`, `COLLAPSE`). v0 target: ~5 000 clips × ~5 tasks = ~25 k examples.

### Task suite
T1 caption, T2 semantic QA, T3 compare, T4 anomaly, T5 explain change, T6 suggest intervention. T6 is the headline evaluable task.

### Model stack (planned)
- Trained encoders: small CNN (or adapter over frozen small CLAP) on mel spectrograms; small transformer on per-node state; topology as concat tokens (no GNN in v0).
- Frozen base LM (Qwen2.5-0.5B or Llama-3.2-1B) + LoRA.
- SFT only for v0; auxiliary losses only if SFT plateaus.

### Evaluation tiers
L1 label accuracy → L2 noise / perturbation robustness → L3 intervention grounding against random + heuristic baselines. Generalization probes across `N`, voice palette, held-out regime.

### Out of scope for v0
Real-time interaction, audio-rate tone coupling, GNNs, RLHF, third-party music datasets, human listening studies.

### Implementation contracts (frozen shape, no engine code yet)
- Simulator contract — `DESIGN_V0.md §9`. Defines scene vs run, canonical YAML, per-step export schema (`state.npz`), audio rendering, event schema, output artifact layout, determinism rules, derived-state helpers, and the `sim.run` / `sim.ablate` / `sim.detect` surface.
- Detector contract — `DESIGN_V0.md §10`. Fair-game vs privileged vs eval-only input boundary; output schema; confidence = evidence margin; thresholds in versioned YAML; counterfactual detectors only via `sim.ablate`.
- Per-label reference — `ONTOLOGY.md` at repo root.

### First implemented artifacts
- `sim/config.py` — config loader + validator; shared by the validator CLI and the run pipeline. Executable event set: `setK`, `nudge`, `impulse`, `ablate_node`. `move` is schema-rejected pending `pos_t` export.
- `sim/garden.py` — real distance-weighted Kuramoto integrator per §2.2 (explicit Euler, SDE noise). Amplitude and audio remain stubbed to constant 1.0 and silent-but-format-correct WAV.
- `sim/derived.py` — pure helpers over exported state per §9.8: `kuramoto_order`, `local_kuramoto_order`, `mean_phase_velocity`, `cluster_assignments` (k-way velocity-gap partition returning a `ClusterAssignments` NamedTuple), `sustained_windows` (find half-open frame-index windows where a 1D bool condition holds for a minimum span, with gap-tolerance hysteresis), `pulse_times_of` (per-node ordered pulse frame indices from the `(T, N)` `pulse_fired` mask — the rhythm-facing projection the future `phase_beating` / `flam` / `polyrhythmic` detectors will all consume). No thresholds, no label output — state-derivation primitives only, shared by tests today and the detector layer tomorrow.
- `sim/__init__.py` — re-exports `load`, `validate`, `simulate`, `ConfigError`, `SCHEMA_VERSION`. Helpers stay in `sim.derived` (explicit import) to keep the package surface narrow.
- `scripts/validate_config.py` — thin CLI wrapping `sim.config.load`.
- `scripts/run_sim.py` — run one config end-to-end; writes the full §9.6 artifact set to `--out`. With `--summary` prints a compact human-readable regime summary (detector verdicts + confidence + supporting `r(t)` and 2-way velocity-separability stats) — the first terminal-demoable regime oracle. With `--summary-json` also writes `summary.json` into the output directory (schema: top-level `schema_version` / `meta` / `detectors` / `stats`; keys sorted, `allow_nan=False`, non-finite separability → `null`). `stats` now carries two compact browser-facing traces in addition to the scalar stats: `r_series = {fps, values}` (the global Kuramoto order parameter `r(t)` downsampled to ~200 points, values rounded to 4 decimal places) and `pulse_raster` (per-node pulse times in seconds, one list per node in node-index order, 4 dp). Both are purely additive under `schema_version 1` (existing readers ignore unknown keys). Both views are rendered from one shared `_build_summary` dict so terminal and JSON cannot drift.
- `sim/detectors.py` — first semantic-detector layer. Five detectors live here: `detect_phase_locked` (sustained `r >= 0.9` for >= 1.0 s, hyst 0.1 s), `detect_drifting` (sustained `r < 0.5` for >= 2.0 s, hyst 0.25 s), `detect_phase_beating` (pairwise — near-frequency pairs with `|Δω̄| ∈ [0.25, 1.5]` rad/s whose unwrapped partner-phase sampled at the reference node's pulse times spans `>= 2π` rad across the analysis window and passes a `R² >= 0.9` linearity test), `detect_flam` (pairwise — frequency-locked pairs with `|Δω̄| <= 0.25` rad/s whose pulse times repeatedly land 15–50 ms apart, `>= 6` times spanning `>= 2` s, with the load-bearing isolation rule that no non-pair node fires within `FLAM_ISOLATION_MARGIN_S = 15` ms of either pair pulse), and `detect_dominant_cluster` (cluster-structure — over a 3 s tail window, requires a `cluster_assignments(v_tail, n_clusters=2).separability_ratio >= 5.0` clean 2-way velocity split with both clusters of size `>= 2` and at least one cluster's tail-window `local_r >= 0.9`). Thresholds and window settings are module-level constants (temporary first-detector policy — YAML thresholds per `DESIGN_V0.md §10.7` are deferred). All five return a uniform `DetectionResult` NamedTuple (`fired`, `windows`, `confidence`, `longest_window_frames`); confidence is evidence margin per §3 doctrine — mean `(r − threshold)` / `(threshold − r)` for the global-coherence detectors, mean `(R² − 0.9)` across qualifying pairs for `phase_beating`, mean `(n_events − 6) / 6` across qualifying pairs for `flam`, mean `(local_r − 0.9)` across qualifying clusters for `dominant_cluster`, with `0.0` when the detector does not fire. Pair/cluster-level debug detail is available via private `_phase_beating_pair_evidence`, `_flam_pair_evidence`, and `_dominant_cluster_evidence` helpers for tests; the `DetectionResult` itself does not carry per-pair/per-cluster metadata. No registry, no label schema, no YAML loading.
- `tests/test_regimes.py` — automated regime smoke tests (one function per regime fixture — `locked`, `two_cluster`, `drifting`). Runs each config through `sim.simulate` into a tempdir and asserts a simple numerical signature of the regime label. Stdlib-only; runnable via `python tests/test_regimes.py` or `pytest`.
- `tests/test_derived.py` — unit tests for pure `sim.derived` helpers that don't need a sim run. First inhabitant: `pulse_times_of` (normal multi-node extraction, empty / no-pulse / `T == 0` / `N == 0` cases, strict-ascending monotonicity, dtype / shape contract, 1D-and-3D rejection).
- `tests/test_detectors.py` — detector-level tests. One function per detector, each exercising the detector across all available regime fixtures and asserting the fire/silent matrix. Observed: `phase_locked` fires only on `regime_locked` (confidence 0.094, 5.73 s window); `drifting` fires only on `regime_drifting` (confidence 0.184, 8.00 s window); `phase_beating` fires only on `regime_phase_beating` (confidence 0.098, pair (0,1) at |Δω̄|=1.223 rad/s, beat ~5.14 s, R²=0.998); `flam` fires only on `regime_flam` (confidence 0.833, pair (0,1) at |Δω̄|=0.017 rad/s, median offset 35.0 ms, 11 isolated events); `dominant_cluster` fires only on `regime_two_cluster` (confidence 0.093, 4+4 split at local_r ≈ 0.993 each, separability ≈ 706); every detector silent on the regimes it does not own. Stdlib-only, same run pattern as `test_regimes.py`.
- `configs/regime_drifting.yaml` — DRIFTING reference (low K0, drifted behaviour; test asserts the 8 s run contains a ≥ 3 s sustained low-coherence window (`r < 0.5`, 0.25 s hysteresis) and tail mean `r < 0.40`; observed tail mean `r ≈ 0.29`, single merged sustained window spanning the full 8 s).
- `configs/regime_locked.yaml` — LOCKED reference; test asserts last-1s `r > 0.9`, observed `r = 0.9953`.
- `configs/regime_two_cluster.yaml` — TWO_CLUSTER reference; test asserts middle-band global `r` + bimodal velocity split + intra-cluster coherence. Observed tail mean `r = 0.628`, clean 4+4 split with bimodal ratio ~700.
- `configs/regime_phase_beating.yaml` — PHASE_BEATING reference. Two near-frequency nodes (2.00, 2.20 Hz) positioned close together with weak `K0=0.30` / `σ=0.15` coupling (effective pair coupling ~0.15 rad/s, well below the |Δω|/2 ≈ 0.63 rad/s two-oscillator Kuramoto lock threshold); six distractor nodes at well-separated frequencies (4.0–7.0 Hz, all adjacent gaps ≥ 0.5 Hz) in a far corner of the unit square so no other pair enters the near-frequency candidate band. `eta=0`, 12 s duration covers ~2 full beat cycles. Exists specifically to exercise `detect_phase_beating` with a positive case — the three earlier fixtures are all negatives by construction.
- `configs/regime_flam.yaml` — FLAM reference (N=4). Two near-frequency nodes (2.00, 2.20 Hz) at `pos=(0.40,0.50)/(0.48,0.50)` with `K0=2.50` / `σ=0.15` (effective pair coupling ~1.47 rad/s — above the |Δω|/2 ≈ 0.63 rad/s two-oscillator lock threshold) so the pair frequency-locks with a fixed phase offset Δθ ≈ 0.44 rad → ~33 ms between pulses, squarely inside the 15–50 ms flam band. Two distractor nodes at 4.0 and 5.0 Hz in the far corner (`x=0.90`) so no other pair enters the velocity band and the per-event isolation filter admits ~50 % of pair events. `eta=0`, 12 s duration covers ~23 pair pulses (~11 after isolation filtering). Exists specifically to exercise `detect_flam` with a positive case — the four earlier fixtures are all negatives by construction.
- `requirements.txt` — `numpy>=2.0`, `pyyaml>=6.0`.
- `.gitignore` — excludes `runs/` outputs and Python bytecode.
- `run.sh` — repo-root convenience launcher for the browser demo. Bash (not strict POSIX — uses an array for the fixture list); `set -euo pipefail`. Creates `.venv` if missing via `python3 -m venv`, installs `requirements.txt` into it only when `import numpy, yaml` fails (cheap pre-flight), then regenerates every fixture's demo artifacts under `runs/demo/<fixture>/` by invoking `scripts/run_sim.py --config configs/regime_<fixture>.yaml --out runs/demo/<fixture> --summary-json` (quiet; fails loudly if any config is missing). Prints a block of copy-pasteable browser URLs (two compelling A/B comparisons — `locked` vs `two_cluster` for the `dominant_cluster` flip, and `two_cluster` vs `phase_beating` for structure-vs-rhythm — plus one single-run URL per fixture), then `exec`s `python -m http.server --bind 127.0.0.1 $PORT` (default `8000`, `PORT=9000 ./run.sh` overrides). URL printing happens before the server starts so the output is visible even while the foreground server blocks. Not a task runner: no Makefile, no dependency-check beyond the venv pre-flight, no log file, no daemonization — a 90-line launcher, explicit in `run.sh`.
- `demo/` — browser-facing surface, A/B comparison mode. `index.html` + `style.css` + `app.js`, vanilla, no build step, no framework, no server-framework. Two independent slots (A and B), each with its own summary file picker, a secondary `+ topology.json` file picker (for `file://` use where siblings can't be auto-fetched), a paste-box, and a status line; query params `?summaryA=URL` and `?summaryB=URL` (legacy `?summary=URL` still loads into slot A) auto-populate when the repo is served by `python -m http.server`, and each slot also best-effort-fetches a sibling `topology.json` next to the summary URL (overridable via `?topologyA=URL` / `?topologyB=URL`). Each slot renders three data cards: (1) a **topology card** (a padded `100×100` SVG viewBox with faint frame + centerline guides, one `<circle>` per node at its unit-square `pos` — y-flipped so config `y=0` sits at the bottom — with a monospace index label and a subtle green-to-blue HSL fill ramp keyed on each node's `omega_0_hz` within the run's frequency range, plus an `N · K₀ · σ · η` footer; gracefully shows "topology unavailable" centered in the stage when no topology is loaded); (2) a **dynamics card** — two stacked `240×{40,N·4.5}` SVG panels showing the `r(t)` sparkline and a per-node pulse raster, fed by `summary.json.stats.r_series` / `stats.pulse_raster`. The sparkline draws a dashed `r = 0.9` reference line (the `phase_locked` threshold) and every fired detector's `windows_s` intervals are painted as translucent green accent bands behind BOTH the sparkline and the raster — so the reader sees at a glance *when* each named regime held. Footer shows `window / lock line / pulses`; (3) the **detector cards** — one per detector (fired vs silent, confidence, longest window, window intervals) — plus a small stats block (`mean r(t)`, `tail 1s r`, optional `tail 2-way velocity separability` with `null` → `"inf"`) and a raw-JSON expander. When both slots are loaded, a **comparison card** appears below them with a rule-based one-line takeaway (e.g. *"A dominant cluster formed; lock collapsed."*), an optional `same topology` / `different topology` badge (per-index position compare with `EPS = 1e-3` when both topologies are loaded), and a bulleted diff list: detector flips (silent ↔ FIRED), confidence changes on pairs that stay fired, and stat deltas (`mean_r`, `tail_1s_mean_r`, separability) with signed values and up/down color accents. The comparison is the semantic focus; the topology card is the static body the semantics live in; the dynamics card is the live physics those semantics summarize. Robust to schema-version mismatches (warns and renders best-effort), missing detectors (skipped in a slot, shown as silent→absent in the diff), missing optional stats / series (panel renders `"r(t) unavailable"` / `"pulse raster unavailable"` placeholders), `null` separability (renders `"inf"`, treated as not-a-number for delta arithmetic), and missing / malformed topology (falls back to the placeholder without blocking any other card).
- `docs/screenshots/` — committed hero screenshots for the README and for remote review when the author is away from the laptop. `hero-locked-vs-two_cluster.png` (A/B showing the `dominant_cluster` detector flip — r(t) pinned near 1 vs oscillating beating pattern) and `structure-vs-rhythm.png` (two_cluster vs phase_beating — same 4+4 body but a different dynamics signature). Regenerated via Chrome headless at viewport `1440×2000`: `"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless=new --disable-gpu --hide-scrollbars --window-size=1440,2000 --virtual-time-budget=3000 --screenshot=docs/screenshots/NAME.png URL` after `run.sh` has started the static server. Committed binaries on purpose — small (~200 KB each) and the README reference is load-bearing for a reader's first-impression signal.

- **`summary.json` schema — additive, no version bump.** Added `stats.r_series = {"fps": float, "values": [float]}` and `stats.pulse_raster = [[float, ...], ...]`. Existing readers ignore unknown keys (the browser demo explicitly names which stats it renders in the stats block, so the new arrays don't leak there). `schema_version` stays at `1`. Sizes on the committed fixtures: `summary.json` grows from ~1.8 KB to ~5–8 KB depending on fixture — trivial on the wire, well below any practical cap.
- **Downsampling `r(t)` — constant target-point count, not constant stride.** Picked `R_SERIES_TARGET_POINTS = 200` and `stride = max(1, T // 200)`. For the 6 s fixtures at 200 Hz this gives stride 6 (→ 201 points at 33.3 fps); for the 12 s fixtures it gives stride 12 (→ 201 points at 16.7 fps). The constant point count is the right optimization target for a browser sparkline (the `<polyline>` needs roughly the same number of vertices regardless of run length); the fps varies and is emitted alongside so readers can reconstruct timestamps. Rounded to 4 decimal places — `r ∈ [0,1]` at 1e-4 precision is invisible on a 240×40 pixel sparkline and halves the JSON size vs raw float64.
- **Pulse raster shape — per-node ordered list of seconds, 4 dp.** Already had `pulse_times_of` in `sim.derived`; the summary builder just wraps it, divides by `control_rate_hz`, and rounds. 4 dp = 100 µs, well below the detector's sub-millisecond timing scales (the tightest threshold is `FLAM_MIN_OFFSET_S = 15 ms`). Alternative shape considered: flat `[[node, t], [node, t], ...]` sorted by time. Rejected because the natural rendering is per-row-per-node, and the nested form makes `raster[i]` directly plottable without a groupby.
- **Firing-window bands are the UX glue between "detector" and "physics".** Every detector that fires emits `windows_s` (half-open seconds intervals) in `summary.json`. The dynamics card walks every detector's windows and draws translucent accent rectangles behind the sparkline AND the raster at those x-ranges. So the reader sees: "r(t) stayed above 0.9 during the green band → that's what `phase_locked: FIRED` was about", and "two_cluster's green band on the right half is the 3s tail that `dominant_cluster` declared". The bands make the detector's claim spatially concrete on the physics.
- **Dashed `r = 0.9` lock line.** The one reference value worth drawing: it's the `phase_locked` gate, the same bar `detect_dominant_cluster` reuses for "at least one cluster coherent enough to stand out", and the first number a reader familiar with Kuramoto physics would look for. Other candidates (`R_DRIFT = 0.5`, `R_LOCK = 0.9`) would clutter; one reference line is enough.
- **SVG-only rendering, no canvas, no library.** ~30 elements per sparkline (one polyline + one reference line + one or two bands), ~N · 20 elements per raster (rows + ticks). Cheap to render and zero build / bundler overhead. Inline `<svg>` with `preserveAspectRatio="none"` so the panels stretch to the stage width while the viewBox stays fixed.
- **Aesthetic decisions.** (a) Accent color = the same green as `var(--accent)` used for FIRED detectors, at 10 % opacity for the window bands and full opacity for the sparkline/raster ticks. Reads as "all-one-story": green = the oracle firing, regardless of surface. (b) Label column on the left (`r(t)`, `pulses`) so the two panels align visually. (c) Background `#0d1017` (same as topology stage) for continuity. (d) Tick `stroke-width = 0.55` SVG units → crisp at the stretched rendering. Tuned by eye on the committed screenshots.

### 2026-04-20 — Claude Opus 4.7
Initial design pass for v0. Key pivots made while drafting:

- **Pulse-driven audio over audio-rate tone coupling.** The obvious starting assumption was continuous tones with Hz-scale beating. Chose pulses instead because (a) it's cheaper to simulate, (b) emergent *rhythm* is the strongest musical phenomenon at the timescale of short clips, and (c) beating can be approximated at the perceptual level by tight near-unison pulses (the `flam` label). Audio-rate tone coupling reconsidered in v0.1.
- **No GNN in v0.** Tempting but unjustified at `N = 8` with known geometry. Concat tokens with positional embeds are sufficient. Revisit only if topology reasoning visibly under-performs.

### 2026-04-20 — Claude Opus 4.7 (doc pass 10 — k-way cluster-assignment primitive)
Generalized the k=2 `split_by_largest_velocity_gap` into a proper reusable k-way partitioner and removed the now-redundant special case. The helper stays narrow: 1D velocity-gap split only, no phase-based sub-clustering, no auto-selection of `k`. Still no detectors, no thresholds YAML, no label schema.

- **`cluster_assignments(mean_vel, n_clusters) -> ClusterAssignments` in `sim/derived.py`.** Given a `(N,)` array of per-node mean phase velocities and a target `n_clusters ∈ [1, N]`, sorts nodes by velocity and splits at the `n_clusters - 1` largest adjacent gaps. Returns a `ClusterAssignments` NamedTuple with three fields: `labels` (`(N,)` int, cluster id per node), `members` (k-tuple of int arrays sorted by node index), and `separability_ratio` (`min(selected_gaps) / max(unselected_gaps)`, with `+inf` / `0.0` at the obvious degenerate boundaries). Clusters are **ordered ascending by mean-velocity centroid** — `members[0]` is the slowest group, `members[k-1]` the fastest — so cluster identity is stable across calls on the same input.
- **Signal chosen: per-node mean phase velocity.** At `N = 8` with coupled-oscillator physics, coherent groups share a common collective rotation rate, so a 1D gap-based split on mean velocity is the honest minimum. Alternatives considered and rejected: (a) phase-only clustering — fails when clusters have different rotation rates (the `two_cluster` case); (b) full feature-vector k-means over `(velocity, sin θ̄, cos θ̄)` — pulls in scipy/sklearn-grade machinery for a problem that's demonstrably 1D at the current scale; (c) hierarchical / DBSCAN variants — same overkill objection, and they all still need a distance metric the caller would have to justify. The honest minimum is one scalar per node and a sort. Phase-based sub-clustering inside a velocity-homogeneous group (e.g. anti-phase at the same rotation rate) is left as a deferred extension — no current or near-term detector needs it.
- **Determinism & tie-breaking.** Node sorting uses `np.argsort(kind="stable")` so equal velocities preserve node-index order. Gap selection uses `np.argsort(-gaps, kind="stable")` then takes the first `n_clusters - 1` indices — stable descending sort means equal-magnitude gaps break ties in favor of the smaller sorted-velocity position (the earlier / slower boundary wins). Near-identical gaps are, as with any gap heuristic, sensitive to input precision — this is documented and intrinsic, not a bug to design around. `separability_ratio` is `+inf` when `n_clusters == 1` (no boundaries to weaken) or when `n_clusters == N` and any selected gap is positive, and `0.0` when all velocities coincide.
- **Removed `split_by_largest_velocity_gap` and `VelocitySplit`.** The k=2 case is now `cluster_assignments(v, 2)`: `members[0]` replaces `.low_indices`, `members[1]` replaces `.high_indices`, `separability_ratio` replaces `.bimodal_ratio` with identical semantics for k=2. The old `gap` field had no callers and is dropped (callers who want it can compute `mean_vel[members[1]].min() - mean_vel[members[0]].max()`). ### 2026-04-20 — Claude Opus 4.7 (doc pass 9 — sustained-window primitive + DRIFTING sanity test)
Closed the simulator sanity triangle (`locked`, `two_cluster`, now `drifting`) and landed the first reusable time-window primitive that sustained-condition detectors (`phase_locked`, `drifting`, `groove`, …) will compose over. No detector framework, no YAML thresholds, no label schema — just a state-level primitive and a regime fixture test.

- **`sustained_windows(condition, min_duration_frames, hysteresis_frames=0)` in `sim/derived.py`.** Takes a 1D bool-castable time series and returns sorted, non-overlapping half-open `[start, end)` frame-index intervals. The interval span (including any absorbed hysteresis gaps) is what gets compared to `min_duration_frames`; the window's endpoints always land on True frames (trailing False hysteresis is never included). `hysteresis_frames=0` disables merging and returns strict contiguous True runs only. Pure function — frame counts only, no config / wall-clock / thresholds. Callers convert seconds to frames themselves via `control_rate_hz`. Edge cases verified in-session: empty input → `[]`, all-False → `[]`, all-True → `[(0, T)]` if it meets `min_duration_frames` else `[]`, 2D input rejected, `min_duration_frames < 1` rejected, `hysteresis_frames < 0` rejected, tiny sub-threshold runs dropped silently, mid-run False gaps `> hysteresis_frames` split into separate windows.
- **Interval convention chosen: half-open `[start, end)`.** Rationale: matches Python slicing (`condition[start:end]` yields exactly the window's frames) and NumPy indexing, and `end - start` gives span directly without off-by-one. The alternative (closed `[start, end]`) would have the nicer "last True index is `end`" property but forces every caller to write `+ 1` for length and for slicing — worse for the three or four upcoming sustained-condition detectors.
- **Hysteresis semantics chosen: gap-absorption at merge time.** Alternatives considered: (a) count only True frames against `min_duration_frames` (rejected — makes "sustained for 3 s" mean "at least 3 s of True out of some larger span", which is a statistical test, not a sustained-condition test); (b) require the condition to hold at both endpoints *and* for all but `hysteresis_frames` interior frames (rejected — harder to implement deterministically, and equivalent in most cases to the span-based rule once `min_duration_frames` is comparable to hysteresis). Current rule: a False stretch between two True runs of length `<= hysteresis_frames` is folded into the window; anything longer splits. Simple, deterministic, and the only interior-False-frame count that matters is the one used for merging.
- **DRIFTING sanity test — `tests/test_regimes.py::test_drifting_regime_sustains_low_coherence`.** Two checks on the 8 s run of `configs/regime_drifting.yaml`: (1) `sustained_windows(r < 0.5, min=2.0 s, hyst=0.25 s)` returns at least one window, with longest `>= 3.0 s`; (2) tail-2 s mean `r < 0.40`, pinning drifting well below two_cluster's middle band (observed `0.63`) and locked's near-unity (observed `0.995`). Observed: one merged sustained window spanning the full 1599 frames (8.00 s); tail mean `r = 0.2902`; fraction of frames with `r < 0.5` = `0.859`.
- **Regime separation is structural, not numeric.** The same `sustained_windows(r < 0.5, min=2.0 s, hyst=0.25 s)` call returns `[]` on both locked (r stays > 0.95, so `low` is never True) and two_cluster (whose longest merged sub-second valley is 86 frames = 0.43 s, well below the 2.0 s floor). Noted in the test's inline comment — the primitive itself is the discriminator, we do not need separate threshold knobs per regime.
- **Existing `configs/regime_drifting.yaml` worked unchanged.** No parameter tuning needed. Tail mean `r ≈ 0.29` on the same seed/config that the earlier doc-pass-6 cross-check measured at `r ≈ 0.27`; the small difference is the different tail length (2 s vs 1 s) used for averaging, not a simulator change — `state.npz` output is still byte-identical run-to-run.

### 2026-04-21 — Claude Opus 4.7 (doc pass 15 — rhythm-side derived primitive)
First rhythm-facing entry in `sim.derived`. `pulse_times_of(pulse_fired) -> tuple[np.ndarray, ...]` turns the `(T, N)` boolean pulse mask into one ascending `int64` frame-index sequence per node. No pairing, no intervals, no ratios, no thresholds — just the raw substrate that the still-unimplemented `phase_beating`, `flam`, and `polyrhythmic` detectors will all consume. Detector layer still 2/13; this pass deliberately does not move that number.

- **Shape kept to the brief's preferred form: `tuple[np.ndarray, ...]`, length always `N`.** Alternatives considered and rejected: (a) ragged 2D padded with `-1` — forces callers to carry the sentinel everywhere and plays poorly with `np.diff`; (b) a flat `(event_count, 2)` array of `(frame, node)` — compact but forces every rhythm detector to re-split by node as its first step; (c) `dict[int, np.ndarray]` — saves a tuple slot for empty nodes but loses the "per-node presence guaranteed" invariant and reads worse at the REPL. The tuple-of-arrays form is what per-node pairwise detectors (`for i, j in combinations(range(N), 2): ...`) already want to iterate over, and empty-but-present entries mean no detector has to special-case "node i has zero pulses" with an `in` check.
- **Units pinned: frame indices, not seconds, not audio samples.** Same unit policy as `sustained_windows` — the helper stays config-free, and callers convert via `frames / control_rate_hz` when they need seconds. Rationale: (a) detector thresholds that matter for rhythm (flam window `10–40 ms`, phase-beating `Δf < 0.5 Hz`) are expressed in seconds, but those thresholds live with each detector, not with the primitive; (b) audio-sample indexing uses `audio_rate_hz` which has a different relationship to the state stream (22050 Hz vs 200 Hz) — collapsing either conversion into `pulse_times_of` would bake in a wrong assumption for the other use site. The primitive is the frame-index substrate; unit conversions are one multiplication away at the call site.
- **`DESIGN_V0.md §9.8` sketch intentionally diverged from.** The spec text lists `pulse_times_of(pulse_fired, t, i) → array of float seconds` — per-call single-node, seconds-valued, carrying the `t` array. Implemented as `pulse_times_of(pulse_fired) → tuple[np.ndarray, ...]` — whole-mask, frame-valued, no `t` needed. Reasoning: every detector that reads pulse times is inherently pairwise or all-vs-all over nodes (phase-beating pairs, flam pair-within-window, polyrhythmic multi-stream), so the per-node loop lives naturally at the helper boundary rather than inside every detector. And the `t` parameter is redundant once callers accept that frame indices are the unit — seconds conversion is `frames * dt` or `frames / control_rate_hz`, trivially one line at the consumer. The §9.8 text is a sketch, not a binding contract; deferring to the cleaner shape is cheaper than pre-committing and later renaming.
- **No companion helper landed.** Considered `inter_pulse_intervals(times)` and `pulse_times_seconds(pulse_fired, dt)`. Rejected both: intervals are a single `np.diff` call at the consumer and "the diff of a pulse-time sequence" is not a rhythm-ontology commitment worth naming; seconds conversion is a single multiplication. The brief explicitly asked for one clean primitive over early rhythm ontology, and every additional helper now pins a semantic the detector layer hasn't needed yet. The bar for a second helper is "two independent detectors both want it in the same shape"; today we have zero detectors that want it at all.
- **Dtype: `int64`, not `intp`.** `np.flatnonzero` returns `intp` (64-bit on this dev machine, 32-bit on 32-bit platforms). Forcing `int64` via `.astype(np.int64, copy=False)` makes the contract platform-independent and the test's `arr.dtype == np.int64` assertion stable across environments. The `copy=False` means no-op on 64-bit platforms (the common case).
- **Edge-case matrix exercised in `tests/test_derived.py`.** `T == 0` (every node present as empty `int64`), `N == 0` (empty tuple), all-False input (every node empty), all-True input on `(6, 2)` (every node `arange(6)`), node with zero pulses surrounded by nodes with pulses (present as empty `int64`, not `None` or omitted), int-castable non-bool input (`np.uint8`) accepted, 1D and 3D input rejected with `ValueError`. Tests are stdlib-only and run standalone without `pytest` just like `test_regimes.py` and `test_detectors.py`.
- **Smoke-checked on a real run.** On `regime_locked`'s 6 s / `T=1201` / `N=8` output: each node fires 13–14 times at the ~2.2 Hz near-consensus rate, all sequences strictly ascending, all `int64`. First pulse frames (60–70) correspond to the initial transient settling into lock — the detector layer will ignore the leading pulse before it's stable.
- **Not done in this pass.** No detectors (`phase_beating`, `flam`, `polyrhythmic` stay unimplemented — they need pairing logic, frequency estimates, window policies that this primitive doesn't prescribe). No intervals helper. No seconds conversion helper. No changes to `summary.json` schema, terminal oracle, or browser demo — the new primitive has no UI surface yet. No update to `DESIGN_V0.md §9.8`'s sketch signature; that text is acknowledged as the spec-at-time-of-writing and the working helper is documented in its own docstring plus this entry.

- **Detector shape: pairwise, one global window.** For each unordered pair `(i, j)` with `|Δω̄| ∈ [0.25, 1.5]` rad/s (the "close but not locked" band), sample `theta[:, j]` at `i`'s pulse frames inside `[warmup, T)`, unwrap the result, and measure two quantities: (a) total coverage `|seq[-1] − seq[0]|` — must be `>= 2π` rad, i.e. one full beat cycle actually completed inside the window; (b) `R²` of a linear fit — must be `>= 0.9`. A pair passes iff both hold. The detector fires iff at least one pair passes. `DetectionResult.windows` is a single analysis-window entry `[(warmup_frames, T)]` — this detector does not carve per-pair sub-windows; the evidence is a property of the whole window, not a transient event.
- **Velocity gate chosen to cleanly separate from the existing regimes.** Empirical survey across the three original fixtures (skipping the 1 s warmup, using the rest as the analysis window): `regime_locked` has every pair at `|Δω̄| ≈ 0` (all nodes tightly co-locked at 13.94 rad/s); `regime_two_cluster` has intra-cluster `|Δω̄| ≈ 0` and inter-cluster `|Δω̄| ≈ 5.58 rad/s` (clean bimodal fingerprint); `regime_drifting` has ten candidate pairs with `|Δω̄| ∈ [1.59, 5.96] rad/s`. Upper gate at 1.5 rad/s excludes the drifting minimum by `0.09` rad/s (about `7×` the `eta=0.03` noise-driven standard error over 8 s — a safe margin). Lower gate at 0.25 rad/s excludes the locked pairs by the width of the lower band. Both gates are round numbers; neither sits on a tuning knife-edge.
- **Drifting is silent, by design, even though it has structured pairwise drift.** `regime_drifting`'s candidate pair (1, 7) at `|Δω̄| = 1.59 rad/s` produces a textbook linear phase-difference drift with `R² = 1.000` and `~1.5` cycles of coverage across the run. Geometrically it IS beating. But "close in mean phase velocity" is the detector's load-bearing phrase, and 1.59 rad/s — a `12.8%` relative frequency spread — is on the outside of what this pass chooses to call "close". The alternative interpretation (fire on all near-linear phase drift regardless of magnitude) would make `phase_beating` and `drifting` partially redundant on the same fixture. Keeping the upper gate tight preserves a clean one-regime-one-detector picture at the cost of requiring a dedicated fixture for the positive case.
- **Two_cluster is silent, by design.** Intra-cluster pairs are locked (`|Δω̄| ≈ 0`, below the lower gate). Inter-cluster pairs have `|Δω̄| ≈ 5.58 rad/s`, beat period `~1.12 s` — fast enough to arguably count as structured drift but far above the upper gate. Either gate alone would silence two_cluster; together they form a comfortable moat around the 0.25–1.5 rad/s "close but not locked" band.
- **`configs/regime_phase_beating.yaml` added as the dedicated positive fixture.** Two nodes at 2.00 and 2.20 Hz (Δω ≈ 1.257 rad/s intrinsic; observed 1.223 rad/s after weak coupling pulls them slightly toward each other) sit close together in the unit square; six distractor nodes at 4.0, 4.5, 5.0, 5.5, 6.0, 7.0 Hz are clustered in the far corner (adjacent `|Δf|` gaps all `>= 0.5 Hz ≈ 3.14 rad/s`, far above the detector's upper gate, so no distractor pair enters the candidate band, and distractor-to-beating-pair Δf is `>= 1.8 Hz`, far above). `K0 = 0.30 rad/s` with `σ = 0.15` gives the beating pair an effective coupling of `~0.15 rad/s` — below the `|Δω|/2 ≈ 0.63 rad/s` two-oscillator Kuramoto lock threshold, so the pair drifts rather than locks. Duration 12 s covers ~2.4 full beat cycles at the observed `~5.14 s` beat period. `eta = 0` keeps the beating signature deterministic. Measured outcome: detector fires with confidence 0.098 (R² = 0.998 against a 0.9 floor → margin ~0.1), exactly one qualifying pair `(0, 1)`.
- **Pair metadata via a private helper, not a new `DetectionResult` field.** `_phase_beating_pair_evidence(theta, phase_vel, pulse_fired, control_rate_hz)` returns a list of dicts (`i, j, dw, slope, r2, coverage_rad, n_pulses_ref`) — one per qualifying pair. Tests consume it directly to assert that exactly the `(0, 1)` pair contributed to the beating fixture's fire. The detector itself wraps the helper and collapses the list into the uniform `DetectionResult` shape. Chosen over extending `DetectionResult` with an optional trailing field because (a) the brief was explicit about not broadly redesigning the detector result system, and (b) pair detail is debug-shaped, not downstream-consumer-shaped — a future `labels.json` writer doesn't want to serialize it.
- **Confidence formula: mean `(R² − 0.9)` across qualifying pairs.** Scale `[0, 0.1]` in practice (`R²` saturates near 1 for clean beating). Parallels `detect_phase_locked`'s `mean(r − R_LOCK)` — both are "how far inside the defining region was the evidence, averaged over what qualified". Alternative considered and rejected: confidence as the ratio of qualifying pairs to all near-frequency candidate pairs. Rejected because the numerator-over-denominator interpretation invites the question "what's a 50% confidence?" and the answer ("half the pairs in the velocity band were structured, half weren't") is a different phenomenon from the one this detector is trying to name.
- **Warmup skip = 1.0 s — big enough to outlast `regime_locked`'s transient and cheap everywhere else.** `regime_locked` takes about 0.25 s to reach r ≥ 0.9; the 1.0 s skip is four times that, conservative but not extravagant. Could have derived `phase_vel` from `theta` internally, but the simulator already exports both; requiring the caller to pass all three matches how `state.npz` is structured and keeps the detector a pure function of the exported state. No mutation of any input array, no state-like globals.
