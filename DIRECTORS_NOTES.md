## Current Canon

### Project thesis
Teach a small local language model to ground musician-native semantics (phase locking, groove, brittleness, interference, etc.) in a bounded synthetic world where those semantics are computable from state. Evaluate on **intervention grounding** — does the model's suggested action actually move the sim the way it claimed?

### Toy world — Coupled Oscillator Garden (v0)
- `N = 8` nodes in the unit square.
- Kuramoto-style phase coupling with distance kernel `Kᵢⱼ = K₀ · exp(−dᵢⱼ/σ)`.
- Optional white-noise perturbation `ξᵢ(t)` with amplitude `η`.
- Per-node fields: `pos, ω₀, θ, A, γ, voice`.
- Audio is **pulse-driven**: each phase-zero crossing fires a percussive grain with a fixed-pitch voice; stereo pan from `pos.x`. Audio-rate tone coupling is deferred to v0.1.
- Scheduled (not real-time) events: `nudge`, `impulse`, `move`, `setK`.

### Run exports
`audio.wav` (22050 Hz stereo), `state.npz` (`θ`, `A` at 200 Hz), `events.jsonl`, `topology.json`, `labels.json`.

### Ontology (13 labels, multi-label, detector-grounded)
`phase_locked`, `drifting`, `polyrhythmic`, `phase_beating`, `dominant_cluster`, `brittle_lock`, `unstable_bridge`, `groove`, `groove_collapse`, `beat_bloom`, `tension_break`, `flam`, `regime_change`. Each has a deterministic detector; labels are ground truth, not model outputs. Detector confidence is **evidence margin** (how strongly / stably the defining condition was met), not epistemic uncertainty.

### Data pipeline
Regime-template sampler (`LOCKED`, `DRIFTING`, `TWO_CLUSTER`, `BRITTLE`, `SWEEP`, `PERTURBED`, `COLLAPSE`). v0 target: ~5 000 clips × ~5 tasks = ~25 k examples.

### Task suite
T1 caption, T2 semantic QA, T3 compare, T4 anomaly, T5 explain change, T6 suggest intervention. T6 is the headline evaluable task.

### Model stack (planned)
- Trained encoders: small CNN (or adapter over frozen small CLAP) on mel spectrograms; small transformer on per-node state; topology as concat tokens (no GNN in v0).
- Frozen base LM (Qwen2.5-0.5B or Llama-3.2-1B) + LoRA.
- SFT only for v0; auxiliary losses only if SFT plateaus.

### Evaluation tiers
L1 label accuracy → L2 noise / perturbation robustness → L3 intervention grounding against random + heuristic baselines. Generalization probes across `N`, voice palette, held-out regime.

### Out of scope for v0
Real-time interaction, audio-rate tone coupling, GNNs, RLHF, third-party music datasets, human listening studies.

### Implementation contracts (frozen shape, no engine code yet)
- Simulator contract — `DESIGN_V0.md §9`. Defines scene vs run, canonical YAML, per-step export schema (`state.npz`), audio rendering, event schema, output artifact layout, determinism rules, derived-state helpers, and the `sim.run` / `sim.ablate` / `sim.detect` surface.
- Detector contract — `DESIGN_V0.md §10`. Fair-game vs privileged vs eval-only input boundary; output schema; confidence = evidence margin; thresholds in versioned YAML; counterfactual detectors only via `sim.ablate`.
- Per-label reference — `ONTOLOGY.md` at repo root.

### First implemented artifacts
- `sim/config.py` — config loader + validator; shared by the validator CLI and the run pipeline. Executable event set: `setK`, `nudge`, `impulse`, `ablate_node`. `move` is schema-rejected pending `pos_t` export.
- `sim/garden.py` — real distance-weighted Kuramoto integrator per §2.2 (explicit Euler, SDE noise). Amplitude and audio remain stubbed to constant 1.0 and silent-but-format-correct WAV. Accepts two optional runtime hooks for the counterfactual path: `ablated_nodes` (set of node indices) and `omega_offsets` ({node_idx: delta_hz} mapping). For each ablated index, the corresponding row and column of the distance-weighted coupling matrix `K_shape` are zeroed at setup (so `setK` events preserve ablation for free) and `pulse_fired[:, k]` is force-silenced after pulse detection; the node's `theta` continues to integrate under its natural frequency so `state.npz` arrays stay `(T, N)`. For each `omega_offsets` entry, the node's natural frequency is shifted by `delta_hz` from t=0 for the whole run while coupling is fully preserved (the perturbation probes lock robustness, not causal removal). Both kwargs can be used together but the public counterfactual entry points expose them one at a time — `sim.ablate.ablate_node` and `sim.ablate.nudge_node`. Out-of-range indices raise `ValueError`. Callers generally do not pass `ablated_nodes` or `omega_offsets` directly — they go through `sim.ablate`.
- `sim/derived.py` — pure helpers over exported state per §9.8: `kuramoto_order`, `local_kuramoto_order`, `mean_phase_velocity`, `cluster_assignments` (k-way velocity-gap partition returning a `ClusterAssignments` NamedTuple), `sustained_windows` (find half-open frame-index windows where a 1D bool condition holds for a minimum span, with gap-tolerance hysteresis), `pulse_times_of` (per-node ordered pulse frame indices from the `(T, N)` `pulse_fired` mask — the rhythm-facing projection the future `phase_beating` / `flam` / `polyrhythmic` detectors will all consume). No thresholds, no label output — state-derivation primitives only, shared by tests today and the detector layer tomorrow.
- `sim/__init__.py` — re-exports `load`, `validate`, `simulate`, `ablate_node`, `nudge_node`, `ConfigError`, `SCHEMA_VERSION`. Helpers stay in `sim.derived` (explicit import) to keep the package surface narrow.
- `sim/ablate.py` — the project's single, narrow counterfactual (intervention-grounded) run surface per `DESIGN_V0.md §9.9` / §10.1. Exposes two sibling entry points, each wrapping `sim.garden.simulate` with one specific deterministic perturbation and writing a small JSON manifest sibling so the artifact bundle self-documents which intervention produced it. (1) `ablate_node(cfg, out_dir, node, config_path=None)` — "decouple and silence": dispatches to `simulate(..., ablated_nodes={node})` and writes `ablation.json` (`schema_version`, `semantics = "decouple_and_silence"`, `ablated_nodes`, `baseline_config_path`). A v0 node ablation **decouples** the node (zero rows + columns in the coupling matrix for the whole run) **and silences** its pulses (`pulse_fired[:, k] = False`); `theta` keeps integrating at the node's natural frequency so `(T, N)` artifact shapes are preserved. (2) `nudge_node(cfg, out_dir, node, delta_hz, config_path=None)` — "detune in place": dispatches to `simulate(..., omega_offsets={node: delta_hz})` and writes `nudge.json` (`schema_version`, `semantics = "detune_in_place"`, `node`, `delta_hz`, `baseline_config_path`). The node's natural frequency is shifted by `delta_hz` for the whole run while coupling, noise, and every other config parameter are preserved — the network still has every chance to recapture the detuned node, so the surviving coherence measures lock elasticity rather than causal removal. Baseline runs emit neither manifest, so the presence of `ablation.json` or `nudge.json` is itself the marker "this bundle is a counterfactual artifact of kind X". No mid-run ablation, no edge ablation, no topology edits, no combined ablation+nudge — each new counterfactual detector earns one narrow sibling seam here; we do not grow a generic intervention registry. Arbitrary interventions remain deferred.
- `scripts/validate_config.py` — thin CLI wrapping `sim.config.load`.
- `scripts/run_ablation.py` — counterfactual CLI mirror of `scripts/run_sim.py`. Takes `--config PATH --out DIR --node K` (plus optional `--seed`, `--summary`, `--summary-json`), dispatches to `sim.ablate.ablate_node`, and renders the regime summary through the same shared `_build_summary` / `_render_summary_text` / `_write_summary_json` helpers `run_sim.py` uses — so baseline and ablated summaries are on identical code paths and directly comparable. The ablated run dir contains the standard §9.6 artifacts plus `ablation.json`.
- `scripts/run_sim.py` — run one config end-to-end; writes the full §9.6 artifact set to `--out`. With `--summary` prints a compact human-readable regime summary (detector verdicts + confidence + supporting `r(t)` and 2-way velocity-separability stats) — the first terminal-demoable regime oracle. With `--summary-json` also writes `summary.json` into the output directory (schema: top-level `schema_version` / `meta` / `detectors` / `stats`; keys sorted, `allow_nan=False`, non-finite separability → `null`). `stats` carries two compact browser-facing traces in addition to the scalar stats: `r_series = {fps, values}` (the global Kuramoto order parameter `r(t)` downsampled to ~200 points, values rounded to 4 decimal places) and `pulse_raster` (per-node pulse times in seconds, one list per node in node-index order, 4 dp). The `detectors` block now includes both counterfactual detectors. **unstable_bridge**: internally runs O(N) ablations via `sim.ablate.ablate_node` on runs where `dominant_cluster` fires on baseline, silent-short-circuits otherwise; when fired, carries one optional field `bridge_nodes: list[int]` (sorted, ascending). **brittle_lock**: internally runs O(N) frequency-nudge reruns via `sim.ablate.nudge_node` on runs where `phase_locked` fires on baseline, silent-short-circuits otherwise; when fired, carries one optional field `brittle_nodes: list[int]` (sorted, ascending). Both node lists are lifted from their respective private `_*_evidence` helpers so the browser demo can name the implicated nodes without re-running the counterfactual loop. The summary builder calls `_unstable_bridge_evidence` / `_brittle_lock_evidence` directly (not the public `detect_*` entry points) and constructs each standard `DetectionResult` locally from the evidence list — so exactly one counterfactual loop per detector runs per summary. Silent-baseline fixtures still short-circuit to zero counterfactual sims; the per-node fields are omitted entirely on silent runs. All additive under `schema_version 1` (existing readers ignore unknown keys). Both views are rendered from one shared `_build_summary` dict so terminal and JSON cannot drift.
- `sim/detectors.py` — semantic-detector layer. Eight detectors live here. Observational: `detect_phase_locked` (sustained `r >= 0.9` for >= 1.0 s, hyst 0.1 s), `detect_drifting` (sustained `r < 0.5` for >= 2.0 s, hyst 0.25 s), `detect_phase_beating` (pairwise — near-frequency pairs with `|Δω̄| ∈ [0.25, 1.5]` rad/s whose unwrapped partner-phase sampled at the reference node's pulse times spans `>= 2π` rad across the analysis window and passes a `R² >= 0.9` linearity test), `detect_flam` (pairwise — frequency-locked pairs with `|Δω̄| <= 0.25` rad/s whose pulse times repeatedly land 15–50 ms apart, `>= 6` times spanning `>= 2` s, with the load-bearing isolation rule that no non-pair node fires within `FLAM_ISOLATION_MARGIN_S = 15` ms of either pair pulse), `detect_polyrhythmic` (pairwise pulse-rate — pairs whose IPI-based rates satisfy `min_rate/max_rate <= 0.85` and sit within `POLY_RATIO_TOL = 2 %` relative deviation of a small-integer ratio in `POLY_RATIO_SET = ((2, 3), (3, 4))`, sustained across three equal sub-windows each within `POLY_SUB_WINDOW_RATIO_TOL = 4 %` of the same `(p, q)` with `>= 3` pulses per node per sub-window and `>= 8` pulses per node over the whole 1 s-warmup analysis window; depends only on `pulse_fired`), `detect_dominant_cluster` (cluster-structure — over a 3 s tail window, requires a `cluster_assignments(v_tail, n_clusters=2).separability_ratio >= 5.0` clean 2-way velocity split with both clusters of size `>= 2` and at least one cluster's tail-window `local_r >= 0.9`). Counterfactual: `detect_unstable_bridge(cfg, theta, phase_vel, control_rate_hz, *, work_dir=None)` — applicability gated by `detect_dominant_cluster` firing on baseline; then for each baseline cluster (from `cluster_assignments(v_tail, 2)`) of size `>= UNSTABLE_BRIDGE_MIN_CLUSTER_SIZE = 3` with tail-window `local_r >= DOMINANT_CLUSTER_MIN_LOCAL_R = 0.9`, runs `sim.ablate.ablate_node(cfg, node=k)` for each member and flags `k` as a bridge iff the surviving members' tail-window `local_r` drops below the same 0.9 gate; and `detect_brittle_lock(cfg, theta, phase_vel, control_rate_hz, *, work_dir=None)` — applicability gated by `detect_phase_locked` firing on baseline; then for each node in `[0, N)` runs `sim.ablate.nudge_node(cfg, node=k, delta_hz=BRITTLE_LOCK_NUDGE_HZ=0.25)` and flags `k` as brittle iff the nudged run's tail-window mean global `r` falls below `BRITTLE_LOCK_MIN_R = R_LOCK = 0.9` (gate reused from `detect_phase_locked`, crossed in the other direction). The two counterfactual detectors are the only ones that take `cfg` on top of observable state (each re-simulates from the scene spec through one narrow `sim.ablate` sibling). Thresholds and window settings are module-level constants (temporary first-detector policy — YAML thresholds per `DESIGN_V0.md §10.7` are deferred). All eight return a uniform `DetectionResult` NamedTuple (`fired`, `windows`, `confidence`, `longest_window_frames`); confidence is evidence margin per §3 doctrine — mean `(r − threshold)` / `(threshold − r)` for the global-coherence detectors, mean `(R² − 0.9)` across qualifying pairs for `phase_beating`, mean `(n_events − 6) / 6` across qualifying pairs for `flam`, mean `(POLY_RATIO_TOL − ratio_dev)` across qualifying pairs for `polyrhythmic`, mean `(local_r − 0.9)` across qualifying clusters for `dominant_cluster`, mean `(baseline_local_r − ablated_local_r)` across qualifying bridge nodes for `unstable_bridge`, mean `(baseline_r − nudged_r)` across qualifying brittle nodes for `brittle_lock`, with `0.0` when the detector does not fire. Pair/cluster/bridge/brittle-level debug detail is available via private `_phase_beating_pair_evidence`, `_flam_pair_evidence`, `_polyrhythmic_pair_evidence`, `_dominant_cluster_evidence`, `_unstable_bridge_evidence`, and `_brittle_lock_evidence` helpers for tests; the `DetectionResult` itself does not carry per-pair/per-cluster/per-bridge/per-brittle metadata. No registry, no label schema, no YAML loading.

Runs each config through `sim.simulate` into a tempdir and asserts a simple numerical signature of the regime label. Stdlib-only; runnable via `python tests/test_regimes.py` or `pytest`.
- `tests/test_derived.py` — unit tests for pure `sim.derived` helpers that don't need a sim run. First inhabitant: `pulse_times_of` (normal multi-node extraction, empty / no-pulse / `T == 0` / `N == 0` cases, strict-ascending monotonicity, dtype / shape contract, 1D-and-3D rejection).
- `tests/test_ablate.py` — counterfactual smoke tests for both `sim.ablate` sibling entry points. Four functions. Ablation: `test_ablate_node_is_not_a_noop` runs `regime_two_cluster` baseline and ablated (node 0) and asserts (1) `(T, N)` artifact shapes match, (2) ablated node's `pulse_fired` is all-False while baseline fires >0 pulses, (3) the *other* N−1 nodes' `theta` trajectories differ from baseline by `max|Δθ| > 1e-3` rad — the "real intervention, not post-hoc silencing" assertion — and (4) `ablation.json` carries the documented schema and baseline runs do not emit it. `test_ablate_node_determinism` asserts byte-identical `state.npz` across two runs of the same ablation. Nudge: `test_nudge_node_is_not_a_noop` runs `regime_brittle_lock` baseline and nudged (node 4, +0.25 Hz) and asserts (1) `(T, N)` shapes match, (2) the nudged node KEEPS pulsing (coupling preserved — the key distinction from ablation), (3) the nudged node's theta diverges from baseline (it's running at a different natural frequency), (4) OTHER nodes also diverge (coupling carries the perturbation through the network), and (5) `nudge.json` carries the documented schema (`semantics = "detune_in_place"`, `node`, `delta_hz`) while baseline runs emit neither `nudge.json` nor `ablation.json`. `test_nudge_node_determinism` asserts byte-identical `state.npz` across two runs of the same nudge. Observed `max|Δθ| ≈ 6.28 rad` across non-ablated nodes on `regime_two_cluster` after ablating node 0; `max|Δθ_self| ≈ 6.07 rad`, `max|Δθ_other| ≈ 6.19 rad`, `n_pulses ≈ 15` on `regime_brittle_lock` after nudging node 4 by +0.25 Hz.
- `tests/test_detectors.py` — detector-level tests. One function per detector, each exercising the detector across the relevant regime fixtures and asserting the fire/silent matrix. Observed: `phase_locked` fires only on `regime_locked` (confidence 0.094, 5.73 s window); `drifting` fires only on `regime_drifting` (confidence 0.184, 8.00 s window); `phase_beating` fires only on `regime_phase_beating` (confidence 0.098, pair (0,1) at |Δω̄|=1.223 rad/s, beat ~5.14 s, R²=0.998); `flam` fires only on `regime_flam` (confidence 0.833, pair (0,1) at |Δω̄|=0.017 rad/s, median offset 35.0 ms, 11 isolated events); `polyrhythmic` fires only on `regime_polyrhythmic` (confidence 0.018, pair (0,1) at rates 2.00/3.00 Hz in a 2:3 ratio, relative deviation 0.22 %, all three sub-windows within 0.6 %); `dominant_cluster` fires only on `regime_two_cluster` (confidence 0.093, 4+4 split at local_r ≈ 0.993 each, separability ≈ 706); `unstable_bridge` fires only on `regime_unstable_bridge` (confidence 0.334, single bridge node 0 in cluster 0 with baseline local_r 0.928 → ablated local_r 0.595 on cluster members minus bridge); `brittle_lock` fires only on `regime_brittle_lock` (confidence 0.113, three brittle nodes 2, 3, 4 at baseline tail-r 0.940 → nudged tail-r 0.898 / 0.823 / 0.758 under a +0.25 Hz whole-run nudge). Every detector silent on the regimes it does not own, including two honest-negative cases on the ablation-loop path: `unstable_bridge` on `regime_two_cluster` (`dominant_cluster` fires on baseline so the ablation loop runs, but every single-node ablation leaves surviving-cluster local_r >= 0.98 — "no load-bearing node" verdict), and `brittle_lock` on `regime_locked` (`phase_locked` fires on baseline so the nudge loop runs, but every single-node +0.25 Hz nudge leaves tail-r ≥ 0.99 — "lock is robust" verdict). Stdlib-only, same run pattern as `test_regimes.py`.
- `configs/regime_drifting.yaml` — DRIFTING reference (low K0, drifted behaviour; test asserts the 8 s run contains a ≥ 3 s sustained low-coherence window (`r < 0.5`, 0.25 s hysteresis) and tail mean `r < 0.40`; observed tail mean `r ≈ 0.29`, single merged sustained window spanning the full 8 s).
- `configs/regime_locked.yaml` — LOCKED reference; test asserts last-1s `r > 0.9`, observed `r = 0.9953`.
- `configs/regime_two_cluster.yaml` — TWO_CLUSTER reference; test asserts middle-band global `r` + bimodal velocity split + intra-cluster coherence. Observed tail mean `r = 0.628`, clean 4+4 split with bimodal ratio ~700.
- `configs/regime_phase_beating.yaml` — PHASE_BEATING reference. Two near-frequency nodes (2.00, 2.20 Hz) positioned close together with weak `K0=0.30` / `σ=0.15` coupling (effective pair coupling ~0.15 rad/s, well below the |Δω|/2 ≈ 0.63 rad/s two-oscillator Kuramoto lock threshold); six distractor nodes at well-separated frequencies (4.0, 4.5, 5.0, 5.5, 6.3, 7.0 Hz — all adjacent gaps ≥ 0.5 Hz; the 6.0 → 6.3 Hz bump breaks the old set's exact 2:3 (4.0/6.0) and 3:4 (4.5/6.0) intrinsic ratios that would otherwise false-fire on `detect_polyrhythmic`) in a far corner of the unit square so no other pair enters the near-frequency candidate band. `eta=0`, 12 s duration covers ~2 full beat cycles. Exists specifically to exercise `detect_phase_beating` with a positive case — the three earlier fixtures are all negatives by construction.
- `configs/regime_flam.yaml` — FLAM reference (N=4). Two near-frequency nodes (2.00, 2.20 Hz) at `pos=(0.40,0.50)/(0.48,0.50)` with `K0=2.50` / `σ=0.15` (effective pair coupling ~1.47 rad/s — above the |Δω|/2 ≈ 0.63 rad/s two-oscillator lock threshold) so the pair frequency-locks with a fixed phase offset Δθ ≈ 0.44 rad → ~33 ms between pulses, squarely inside the 15–50 ms flam band. Two distractor nodes at 4.0 and 5.0 Hz in the far corner (`x=0.90`) so no other pair enters the velocity band and the per-event isolation filter admits ~50 % of pair events. `eta=0`, 12 s duration covers ~23 pair pulses (~11 after isolation filtering). Exists specifically to exercise `detect_flam` with a positive case — the four earlier fixtures are all negatives by construction.
- `configs/regime_polyrhythmic.yaml` — POLYRHYTHMIC reference (N=4). Two nodes at 2.00 and 3.00 Hz (exact 2:3 intrinsic ratio) at `pos=(0.40,0.50)/(0.48,0.50)` with `K0=0.50` / `σ=0.15` giving effective pair coupling ~0.29 rad/s — well below the |Δω|/2 ≈ 3.14 rad/s two-oscillator lock threshold, so each oscillator runs at its intrinsic rate (mean-rate correction at this K/Δω ratio is O((K/Δω)²) ≈ 0.2 %), preserving the clean 2:3 pulse-rate signature. Two distractor nodes at 4.20 and 4.80 Hz in the far corner (`x=0.90`) chosen so no other pair enters a 2:3 or 3:4 band at the detector's 2 % whole-window tolerance: (4.2, 4.8) ratio 0.875 sits above the near-unity gate; pair-distractor cross pairs all clear the small-ratio band by at least 4.8 %. `eta=0`, 10 s duration gives a 9 s analysis window after the 1 s warmup skip — the slow node fires ~18 pulses, the fast ~27, comfortably above the 8-pulse floor, and each of three sub-windows carries ~6/~9 pulses above the 3-per-sub floor. Exists specifically to exercise `detect_polyrhythmic` with a positive case — the five earlier fixtures are all negatives by construction.
- `configs/regime_unstable_bridge.yaml` — UNSTABLE_BRIDGE reference (N=8). Cluster A is a three-node bridge + two-spoke structure: node 0 at `pos=(0.50, 0.30)` / `ω=2.10 Hz` sits between two geometrically distant spokes (node 1 at `(0.05, 0.10)` / 2.00 Hz and node 2 at `(0.95, 0.10)` / 2.20 Hz, distance ≈ 0.9). With `K0=15.0` / `σ=0.20`, the bridge–spoke links carry `K_eff ≈ 1.28` rad/s at |Δω|=0.63 rad/s (`K/(Δω/2) ≈ 4.1`, solid lock), while the direct spoke–spoke link is `K_eff ≈ 0.17` rad/s at |Δω|=1.26 rad/s (`K/(Δω/2) ≈ 0.27`, far below the pair-lock threshold). With the bridge present the trio locks at baseline `local_r ≈ 0.93`; ablate node 0 and the surviving spokes desynchronize to `local_r ≈ 0.60` over the 3 s tail window. Cluster B (nodes 3–7) is a tightly-packed NE-corner distractor at 3.30–3.50 Hz — chosen to clear the polyrhythmic detector's 2 % ratio tolerance for any A/B cross pair (closest cross-ratio ≈ 0.618, ~7 % off 2:3) while providing enough separability for `dominant_cluster` to fire on baseline (observed `sep ≈ 303`). `detect_flam` co-fires on the bridge–spoke pairs (locked-phase offset ~30 ms is in the 15–50 ms flam band, and each pair is isolated from the distant cluster B by the 15 ms isolation margin) — a legitimate physical co-occurrence, not a misfire. Exists specifically to exercise `detect_unstable_bridge` with a clean one-bridge-node positive case — the six earlier fixtures are all negatives for this detector (five by dominant_cluster precondition, `regime_two_cluster` by honest "no single load-bearing node" verdict).
- `configs/regime_brittle_lock.yaml` — BRITTLE_LOCK reference (N=5). A tight spatial cluster (all pairwise `d ≈ 0.07–0.20` around `(0.5, 0.5)`) at a marginal coupling/ω-spread ratio — `K0=1.00` / `σ=0.15` drives per-node total coupling ~1.6 rad/s against a uniform ω spread of 0.40 Hz (≈ 1.88 rad/s full width, half-width ≈ 1.26 rad/s). The ensemble locks — baseline tail-3s `r ≈ 0.94`, passing `detect_phase_locked` with a thin 0.04 margin — but the lock is marginal enough that a `+0.25 Hz` (`≈ 1.57 rad/s`) single-node whole-run frequency nudge breaks it for the three outer-ω nodes (2, 3, 4 collapse tail-r to 0.898 / 0.823 / 0.758 respectively) while inner-ω nodes (0, 1) stay in the coupling basin and the lock holds. `eta=0`, 6 s duration gives a 3 s tail window that fully resolves the nudged regime. Exists specifically to exercise `detect_brittle_lock` with a clean three-brittle-node positive case — the seven earlier fixtures are all negatives for this detector (six by `phase_locked` precondition; `regime_locked` by honest "lock is robust — every single-node 0.25 Hz nudge leaves tail-r ≥ 0.99" verdict, i.e. it passes the applicability gate but the ensemble absorbs the perturbation).

- `run.sh` — repo-root convenience launcher for the browser demo. Bash (not strict POSIX — uses an array for the fixture list); `set -euo pipefail`. Creates `.venv` if missing via `python3 -m venv`, installs `requirements.txt` into it only when `import numpy, yaml` fails (cheap pre-flight), then regenerates every fixture's demo artifacts (including `regime_unstable_bridge` as of doc pass 25 and `regime_brittle_lock` as of doc pass 27) under `runs/demo/<fixture>/` by invoking `scripts/run_sim.py --config configs/regime_<fixture>.yaml --out runs/demo/<fixture> --summary-json` (quiet; fails loudly if any config is missing). Prints a block of copy-pasteable browser URLs (two compelling A/B comparisons — `locked` vs `two_cluster` for the `dominant_cluster` flip, and `two_cluster` vs `phase_beating` for structure-vs-rhythm — plus the single-run `unstable_bridge` URL that shows the first counterfactual detector firing, plus one single-run URL per fixture), then `exec`s `python -m http.server --bind 127.0.0.1 $PORT` (default `8000`, `PORT=9000 ./run.sh` overrides). URL printing happens before the server starts so the output is visible even while the foreground server blocks. Not a task runner: no Makefile, no dependency-check beyond the venv pre-flight, no log file, no daemonization — a ~95-line launcher, explicit in `run.sh`.

## Resolved Dragons and Pivots

### 2026-04-22 — Claude Opus 4.7 (validation pass — `BRITTLE_LOCK_NUDGE_HZ = 0.25` sweep)

Tiny empirical pressure-test of the `BRITTLE_LOCK_NUDGE_HZ = 0.25` constant introduced in doc pass 27. No new detector, no new `sim.ablate` surface, no demo-layer changes — this is a threshold-audit only. Question asked: is `0.25 Hz` a stable, honest constant or a one-fixture accident? Answer: **stable** — it sits at the low edge of a {0.25, 0.30, 0.35} Hz plateau on both fixtures available. Constant stays canon, with one honest caveat noted.

- **Sweep method.** One-off script at `/tmp/brittle_lock_sweep.py` (not committed — threshold audits are not a persistent tool surface). For each fixture in `{regime_locked, regime_brittle_lock}` and each `delta ∈ {0.15, 0.20, 0.25, 0.30, 0.35}` Hz, ran one baseline simulation and N nudged counterfactuals via `sim.ablate.nudge_node` (N = 8 and 5 respectively). Firing rule mirrors the detector: tail-window (last 3.0 s) mean global Kuramoto `r < 0.9` on the nudged run. No code path exercised that the detector itself doesn't already exercise — this is a configuration audit, not a re-implementation. Cost: 65 nudged sims total, ~4 s wall-time in the venv.
- **Sweep results — `regime_locked` (robust lock, expected silent).**

  | Δ (Hz) | fires? | brittle nodes | worst nudged tail-r | worst drop |
  |--------|--------|---------------|---------------------|-----------|
  | 0.15   | no     | —             | 0.9927              | 0.0025    |
  | 0.20   | no     | —             | 0.9916              | 0.0037    |
  | 0.25   | no     | —             | 0.9902              | 0.0050    |
  | 0.30   | no     | —             | 0.9887              | 0.0066    |
  | 0.35   | no     | —             | 0.9869              | 0.0084    |

  Baseline tail-r = 0.9953. Worst-case nudged tail-r grows smoothly from 0.993 to 0.987 across the sweep; the lock is nowhere near the `R_LOCK = 0.9` gate at any delta tested. There is no cliff inside `[0.15, 0.35]` for this fixture — the headroom is ~0.09 r at 0.25 and still ~0.087 at 0.35.

- **Sweep results — `regime_brittle_lock` (marginal lock, expected fired).**

  | Δ (Hz) | fires? | brittle nodes | worst nudged tail-r | worst drop |
  |--------|--------|---------------|---------------------|-----------|
  | 0.15   | yes    | {4}           | 0.8581              | 0.0814    |
  | 0.20   | yes    | {3, 4}        | 0.7487              | 0.1908    |
  | 0.25   | yes    | {2, 3, 4}     | 0.7580              | 0.1815    |
  | 0.30   | yes    | {2, 3, 4}     | 0.7134              | 0.2262    |
  | 0.35   | yes    | {2, 3, 4}     | 0.6808              | 0.2588    |

  Baseline tail-r = 0.9396. The detector fires at **every** delta in the sweep — the binary `fired` outcome is not sensitive to the exact choice of `BRITTLE_LOCK_NUDGE_HZ` in this range. What is sensitive is the *identity* of the brittle-node set: the outer-ω triplet `{2, 3, 4}` only emerges at `Δ ≥ 0.25`; at `Δ = 0.20` node 2 still holds at r = 0.916 (on the edge), and at `Δ = 0.15` only the outermost node 4 breaks. Above 0.25 the set saturates at `{2, 3, 4}` — 0.30 and 0.35 give the same brittle-node identity, only harder drops.

- **Is 0.25 still the best narrow constant?** Yes, with the caveat below. Argument:
  1. `0.25` is the smallest Δ in the `{0.25, 0.30, 0.35}` plateau where the `regime_brittle_lock` brittle-node set matches the documented physics story (outer-ω triplet, per `configs/regime_brittle_lock.yaml` lines 17–21). Smaller Δ fires but flags a strict subset — a narrower and less informative story.
  2. On `regime_locked` the 0.25 Hz probe leaves every node's tail-r above 0.99 — a ~0.09 r margin above `R_LOCK`. The robust-lock reference is nowhere near misclassification.
  3. Choosing the low edge of the plateau (0.25 rather than 0.30 or 0.35) keeps the intervention "as small as it needs to be" — consistent with the nudge's honest-probe framing: the goal is to measure elasticity, not to guarantee collapse.
  4. The detector's binary `fired` outcome is invariant across the full `[0.15, 0.35]` Hz sweep on both fixtures — meaning the verdict (which is what consumers actually read) is robust to a ~40 % variation in the constant either side of 0.25. Only the `brittle_nodes` extras field is Δ-dependent.

- **Caveat — narrow plateau upper bound is not constrained by current fixtures.** We can show `Δ = 0.25 Hz` sits at the low edge of a `{0.25, 0.30, 0.35}` plateau, but not that it is in the middle. `regime_locked` absorbs the full sweep (worst tail-r = 0.987 at 0.35), so the sweep does not find the Δ at which the robust-lock reference starts to near-miss. A single intermediate-robustness lock fixture (between `regime_locked`'s r ≈ 0.995 and `regime_brittle_lock`'s r ≈ 0.94) would let us bracket the plateau from above. Until we have such a fixture, the honest framing is "0.25 Hz is the smallest value that produces the full documented brittle-node set on the one positive fixture available, and leaves the one robust-lock fixture unambiguously silent" — not "0.25 Hz is the midpoint of a narrow honest band."
- **Caveat — `brittle_nodes` identity is Δ-indexed, not a fixture property.** Downstream consumers (`summary.json.detectors.brittle_lock.brittle_nodes`, any future topology-ring treatment) should read the set as "nodes that break under this specific Δ", not as an intrinsic property of the scene. Flipping `BRITTLE_LOCK_NUDGE_HZ` from 0.20 to 0.25 changes the set from `{3, 4}` to `{2, 3, 4}` on the same fixture. This is expected physics, not a bug — documenting here so that a future "tighten the nudge" impulse doesn't quietly lose node 2 from the narrative.
- **What stays canon.** `BRITTLE_LOCK_NUDGE_HZ = 0.25 Hz` in `sim/detectors.py:1453`. No code or config change from this pass.
- **What is still not done.** No third lock fixture at intermediate robustness (would constrain the plateau's upper bound). No automated regression test over the full Δ sweep (tests still pin only the single `Δ = 0.25` path — adding a sweep as a unit test would slow the suite for modest incremental confidence). No YAML threshold file (per `DESIGN_V0.md §10.7`, deferred across all detectors). The sweep script itself lives at `/tmp/brittle_lock_sweep.py` — not committed; threshold audits are ad-hoc by design.

- **Perturbation contract: whole-run additive ω offset, coupling preserved.** `nudge_node(cfg, out_dir, node, delta_hz, config_path=None)` writes `nudge.json` with `schema_version=1`, `semantics="detune_in_place"`, `node`, `delta_hz`, `baseline_config_path`. Under the hood it calls `simulate(cfg, out, omega_offsets={node: delta_hz})`; the integrator adds `delta_hz` to that node's natural frequency at t=0 and holds it for the whole run. **Coupling stays intact** — the distance-weighted K matrix is unchanged, noise is unchanged, events are unchanged, the RNG stream still derives from `seed` alone. Determinism: `(config + seed + node + delta_hz)` → byte-identical `state.npz` (tested). Every other artifact shape is preserved — `pulse_fired[:, node]` is computed from the detuned dynamics, not force-silenced. This is the critical distinction from ablation: ablation removes the node's *causal contribution*; nudge leaves the contribution intact but changes its natural frequency, so the ensemble gets to try to recapture it. The detector's firing rule keys on whether it succeeds.
- **Why "additive whole-run ω offset" and not the alternatives considered.** Rejected: (a) **frequency nudge ladder** (probe multiple deltas per node to find the critical `delta_c`). More info but O(N·L) sim reruns for a ladder of length L; the brief asked for the smallest honest version, and a single chosen delta is honest because it's the size at which the specific physical question "does this node's small detuning break the lock?" has a binary answer. (b) **mid-run one-shot nudge** (fire a `nudge` event at t=t0). The schema already supports this (via `events[{"type":"nudge",...}]`), but it tangles "the perturbation" with "the transient" — brittle-lock is a steady-state question, cleaner measured with a whole-run offset. (c) **multiplicative scaling of K0 for the node** (simulate weakened coupling). That changes the *coupling basin* per-node, which is a different physical claim than detuning; would ask "is this node's coupling load-bearing?" — a future counterfactual detector's question, not this one's. (d) **sibling bundled with ablate (same function, both kwargs)**. Rejected on narrowness: `ablate_node(k, delta_hz=...)` would conflate two perturbation kinds behind one verb, obscuring the manifest semantics. Two narrow verbs + two manifest kinds read better both in code and in artifact archaeology.
- **Simulator seam: exactly one new kwarg on `simulate`.** `simulate(..., omega_offsets=None)` accepts `{node_idx: delta_hz}` and applies `omega[k] += delta_hz` right after `omega` is initialized. That's the whole runtime patch — nine lines added to `sim/garden.py`. No branches in the integrator loop, no extra artifact path, no schema change. Out-of-range indices raise `ValueError` to match the existing `ablated_nodes` behaviour. `ablated_nodes` and `omega_offsets` can be combined at the simulator level (they live in separate state), but the public `sim.ablate` surface exposes them one at a time — we deliberately do not open up a "cross product" intervention combinator.
- **Detector contract: applicability gate reuses `phase_locked`, breakage gate reuses `R_LOCK`.** `detect_brittle_lock(cfg, theta, phase_vel, control_rate_hz, *, work_dir=None)` fires iff (a) `detect_phase_locked(theta, rate).fired == True` on baseline and (b) there exists some node `k ∈ [0, N)` whose nudged run (via `sim.ablate.nudge_node(cfg, node=k, delta_hz=0.25)`) produces tail-window (last `BRITTLE_LOCK_TAIL_S = 3 s`) mean global `r < BRITTLE_LOCK_MIN_R = R_LOCK = 0.9`. Reusing `R_LOCK` is deliberate: the brittleness claim is "the nudge made the ensemble stop qualifying as phase-locked". One less free parameter, one more invariant across detectors.

- **Semantics chosen: "decouple and silence".** An ablated node (a) has its coupling-matrix row and column zeroed for the whole run, so it neither drives nor is driven by any other node; (b) has `pulse_fired[:, k]` forced all-False so it emits no rhythmic events; and (c) continues to integrate `theta` at its own natural frequency so `state.npz` arrays stay `(T, N)`.     "baseline_config_path": "configs/regime_two_cluster.yaml",

- **Pair-rate ratio, not pair-phase commensurability.** Considered two shapes for the detector signature: (a) fit the cumulative modular phase `ψ = q·θ_i − p·θ_j` to a near-zero slope (mirrors `phase_beating`'s R² linearity test), (b) compare per-node pulse rates against a small-integer ratio table. Shape (a) is strictly stronger information, but in v0 (no amplitude coupling, no mode-lock drag) shapes (a) and (b) give the same verdict: if mean pulse rates are in p:q ratio, the modular phase automatically has zero slope. (b) is cheaper, reads as "the two pulse trains count off in a 2:3 pattern", and stays more legibly pulse-domain honest than "a specific modular combination of two wrapped phases is linear". Picked (b).
- **Rate estimator: IPI, not count-over-window.** `rate = (n_pulses − 1) / (last_pulse_time − first_pulse_time)` instead of `n / window_duration`. The count-over-window form suffers ±1 boundary quantization (~7 % at `n ≈ 14`), which at a 2 % tolerance is the difference between firing and not. IPI-based rate averages exactly across a periodic train — for the polyrhythmic fixture's 2 Hz node over a 3 s sub-window, any of `n = 5, 6, 7` pulses yields the same IPI rate of ~2.00 Hz once you divide the span by `n − 1`.

Dropping `4:5` was the clean call: the detector's claim remains legible ("2:3 and 3:4 are the most rhythmically distinct small polyrhythms") while staying strictly pulse-domain honest on the existing fixtures. - **Detector shape: pairwise, one global window, two hard gates + isolation + sustained-count.** For each unordered pair `(i, j)` with `|Δω̄| <= FLAM_DW_MAX = 0.25 rad/s` (near-locked), scan every pulse in `i` inside the analysis window `[warmup, T)` with warmup `FLAM_WARMUP_S = 1.0 s`, pair it with the nearest pulse in `j`, and keep the pair event iff (a) `FLAM_MIN_OFFSET_S = 0.015 s <= |t_j − t_i| <= FLAM_MAX_OFFSET_S = 0.050 s` (15–50 ms flam band) and (b) no non-pair node has a pulse within `FLAM_ISOLATION_MARGIN_S = 0.015 s` of either pair pulse on either side. The pair qualifies iff it accumulates `>= FLAM_MIN_EVENTS = 6` such events with first-to-last span `>= FLAM_MIN_SPAN_S = 2.0 s`. The detector fires iff `>= 1` pair qualifies. `DetectionResult.windows` is a single analysis-window entry `[(warmup_frames, T)]` — flam is a whole-window pair-level property, not a per-event sub-window.
- **The load-bearing design call: isolation as a padded-window, not strictly-between.** First cut used "no non-pair pulse strictly between `(t_i, t_j)`". It kept `regime_locked` silent (the extreme pair has intermediate nodes firing *inside* the pair interval — caught) but fired on `regime_two_cluster` (its 4-node cluster A fires in order `3 → 2 → 1 → 0` at 5 ms increments; the leading pair `(0, 1)` is 15 ms apart with *nothing* strictly between them — nodes 2 and 3 fire 5 and 10 ms *before* node 1). Replaced strictly-between with a padded-window rule: no non-pair pulse in `[min(t_i, t_j) − margin, max(t_i, t_j) + margin]`. Chose margin = `FLAM_MIN_OFFSET_S = 15 ms` — a clean derivation from an already-tuned constant, semantic is "nothing else firing within a flam-min-offset of the pair". Silences two_cluster; keeps regime_locked silent for the same reason as before; and the `regime_flam` fixture still passes because its two distractors at 4 / 5 Hz leave most pair events with clean ±15 ms buffers.
- **Velocity gate chosen below every near-locked negative and above every drifting positive.** `|Δω̄| <= 0.25 rad/s`. Empirical survey:
  - `regime_locked`: every pair at |Δω̄| ≈ 0 — **under** the gate (as designed; isolation rules it out at the pair-event level).
  - `regime_drifting`: smallest candidate pair |Δω̄| ≈ 1.59 rad/s — **above** the gate, excluded here.
  - `regime_two_cluster`: intra-cluster pairs |Δω̄| ≈ 0 (under, isolation catches); inter-cluster ~5.6 rad/s (above).
  - `regime_phase_beating`: beating pair |Δω̄| ≈ 1.22 rad/s — **above** the gate; honest (beating drifts through all offsets, flam requires a held offset).
  - `regime_flam`: the pair locks at |Δω̄| ≈ 0.017 rad/s — **under** the gate by ~15×, not sitting on a knife-edge.
- **Offset band chosen musically and empirically.** `[15, 50] ms`. Lower bound: `regime_locked`'s adjacent pairs have ~2–10 ms offsets (tight-lock crowd, excluded by band) while its extreme pairs reach ~20 ms (in-band but caught by isolation). Upper bound: 50 ms is the far side of the musical flam — longer than that reads as two distinct hits rather than a near-unison. Keeps the detector from morphing into a polyrhythmic / cross-rhythm claim on longer offsets.
- **Confidence formula: `mean over pairs of (n_events − FLAM_MIN_EVENTS) / FLAM_MIN_EVENTS`.** A "how many times over the minimum did we see" margin. Observed on `regime_flam`: pair (0,1) gave 11 events vs 6 minimum → margin = 0.833. Unbounded upward but in practice `<= ~5` at 12 s run / 2 Hz pair rate. Parallels `detect_phase_locked`'s margin shape ("how far inside the defining region, averaged over firing cases") while living in a different unit — the other detectors' margins are confidence-ish `[0, 0.5]`, flam's is count-ratio `[0, ~5]`. Considered normalizing to `[0, 1]` by capping at some "plenty" count; rejected because the raw ratio is honest and unboundedness only matters if a consumer tries to compare across detectors, which no consumer currently does.

### 2026-04-20 — Claude Opus 4.7 (doc pass 10 — k-way cluster-assignment primitive)
Generalized the k=2 `split_by_largest_velocity_gap` into a proper reusable k-way partitioner and removed the now-redundant special case. The helper stays narrow: 1D velocity-gap split only, no phase-based sub-clustering, no auto-selection of `k`. Still no detectors, no thresholds YAML, no label schema.

- **`cluster_assignments(mean_vel, n_clusters) -> ClusterAssignments` in `sim/derived.py`.** Given a `(N,)` array of per-node mean phase velocities and a target `n_clusters ∈ [1, N]`, sorts nodes by velocity and splits at the `n_clusters - 1` largest adjacent gaps. Returns a `ClusterAssignments` NamedTuple with three fields: `labels` (`(N,)` int, cluster id per node), `members` (k-tuple of int arrays sorted by node index), and `separability_ratio` (`min(selected_gaps) / max(unselected_gaps)`, with `+inf` / `0.0` at the obvious degenerate boundaries). Clusters are **ordered ascending by mean-velocity centroid** — `members[0]` is the slowest group, `members[k-1]` the fastest — so cluster identity is stable across calls on the same input.
- **Signal chosen: per-node mean phase velocity.** At `N = 8` with coupled-oscillator physics, coherent groups share a common collective rotation rate, so a 1D gap-based split on mean velocity is the honest minimum. Alternatives considered and rejected: (a) phase-only clustering — fails when clusters have different rotation rates (the `two_cluster` case); (b) full feature-vector k-means over `(velocity, sin θ̄, cos θ̄)` — pulls in scipy/sklearn-grade machinery for a problem that's demonstrably 1D at the current scale; (c) hierarchical / DBSCAN variants — same overkill objection, and they all still need a distance metric the caller would have to justify. The honest minimum is one scalar per node and a sort. Phase-based sub-clustering inside a velocity-homogeneous group (e.g. anti-phase at the same rotation rate) is left as a deferred extension — no current or near-term detector needs it.
- **Determinism & tie-breaking.** Node sorting uses `np.argsort(kind="stable")` so equal velocities preserve node-index order. Gap selection uses `np.argsort(-gaps, kind="stable")` then takes the first `n_clusters - 1` indices — stable descending sort means equal-magnitude gaps break ties in favor of the smaller sorted-velocity position (the earlier / slower boundary wins). Near-identical gaps are, as with any gap heuristic, sensitive to input precision — this is documented and intrinsic, not a bug to design around. `separability_ratio` is `+inf` when `n_clusters == 1` (no boundaries to weaken) or when `n_clusters == N` and any selected gap is positive, and `0.0` when all velocities coincide.
- **Removed `split_by_largest_velocity_gap` and `VelocitySplit`.** The k=2 case is now `cluster_assignments(v, 2)`: `members[0]` replaces `.low_indices`, `members[1]` replaces `.high_indices`, `separability_ratio` replaces `.bimodal_ratio` with identical semantics for k=2. The old `gap` field had no callers and is dropped (callers who want it can compute `mean_vel[members[1]].min() - mean_vel[members[0]].max()`). No deprecation shim — only one caller existed (the two_cluster test) and it's refactored in the same pass.
- **Test refactor is implementation-only.** `tests/test_regimes.py::test_two_cluster_regime_shows_bimodal_structure` now uses `cluster_assignments(v_per_node, n_clusters=2)` and reads `members`, `separability_ratio`. Variable names `low_indices`/`high_indices` → `slow`/`fast` to match the centroid-ascending ordering convention. All assertions and tolerances unchanged. - **Not done in this pass.** No detector registry. No thresholds YAML. No label schema. No phase-based sub-clustering. No automatic `k` selection (that's a detector-layer concern — sustained-condition detectors will pass their own `k` based on the regime hypothesis under test). No new configs.

### 2026-04-20 — Claude Opus 4.7 (doc pass 9 — sustained-window primitive + DRIFTING sanity test)
Closed the simulator sanity triangle (`locked`, `two_cluster`, now `drifting`) and landed the first reusable time-window primitive that sustained-condition detectors (`phase_locked`, `drifting`, `groove`, …) will compose over. No detector framework, no YAML thresholds, no label schema — just a state-level primitive and a regime fixture test.

- **`sustained_windows(condition, min_duration_frames, hysteresis_frames=0)` in `sim/derived.py`.** Takes a 1D bool-castable time series and returns sorted, non-overlapping half-open `[start, end)` frame-index intervals. The interval span (including any absorbed hysteresis gaps) is what gets compared to `min_duration_frames`; the window's endpoints always land on True frames (trailing False hysteresis is never included). `hysteresis_frames=0` disables merging and returns strict contiguous True runs only. Pure function — frame counts only, no config / wall-clock / thresholds. Callers convert seconds to frames themselves via `control_rate_hz`. Edge cases verified in-session: empty input → `[]`, all-False → `[]`, all-True → `[(0, T)]` if it meets `min_duration_frames` else `[]`, 2D input rejected, `min_duration_frames < 1` rejected, `hysteresis_frames < 0` rejected, tiny sub-threshold runs dropped silently, mid-run False gaps `> hysteresis_frames` split into separate windows.
- **Interval convention chosen: half-open `[start, end)`.** Rationale: matches Python slicing (`condition[start:end]` yields exactly the window's frames) and NumPy indexing, and `end - start` gives span directly without off-by-one. The alternative (closed `[start, end]`) would have the nicer "last True index is `end`" property but forces every caller to write `+ 1` for length and for slicing — worse for the three or four upcoming sustained-condition detectors.
- **Hysteresis semantics chosen: gap-absorption at merge time.** Alternatives considered: (a) count only True frames against `min_duration_frames` (rejected — makes "sustained for 3 s" mean "at least 3 s of True out of some larger span", which is a statistical test, not a sustained-condition test); (b) require the condition to hold at both endpoints *and* for all but `hysteresis_frames` interior frames (rejected — harder to implement deterministically, and equivalent in most cases to the span-based rule once `min_duration_frames` is comparable to hysteresis). Current rule: a False stretch between two True runs of length `<= hysteresis_frames` is folded into the window; anything longer splits. Simple, deterministic, and the only interior-False-frame count that matters is the one used for merging.
- **DRIFTING sanity test — `tests/test_regimes.py::test_drifting_regime_sustains_low_coherence`.** Two checks on the 8 s run of `configs/regime_drifting.yaml`: (1) `sustained_windows(r < 0.5, min=2.0 s, hyst=0.25 s)` returns at least one window, with longest `>= 3.0 s`; (2) tail-2 s mean `r < 0.40`, pinning drifting well below two_cluster's middle band (observed `0.63`) and locked's near-unity (observed `0.995`). Observed: one merged sustained window spanning the full 1599 frames (8.00 s); tail mean `r = 0.2902`; fraction of frames with `r < 0.5` = `0.859`.

### 2026-04-21 — Claude Opus 4.7 (doc pass 15 — rhythm-side derived primitive)
First rhythm-facing entry in `sim.derived`. `pulse_times_of(pulse_fired) -> tuple[np.ndarray, ...]` turns the `(T, N)` boolean pulse mask into one ascending `int64` frame-index sequence per node. No pairing, no intervals, no ratios, no thresholds — just the raw substrate that the still-unimplemented `phase_beating`, `flam`, and `polyrhythmic` detectors will all consume. Detector layer still 2/13; this pass deliberately does not move that number.

- **Shape kept to the brief's preferred form: `tuple[np.ndarray, ...]`, length always `N`.** Alternatives considered and rejected: (a) ragged 2D padded with `-1` — forces callers to carry the sentinel everywhere and plays poorly with `np.diff`; (b) a flat `(event_count, 2)` array of `(frame, node)` — compact but forces every rhythm detector to re-split by node as its first step; (c) `dict[int, np.ndarray]` — saves a tuple slot for empty nodes but loses the "per-node presence guaranteed" invariant and reads worse at the REPL. The tuple-of-arrays form is what per-node pairwise detectors (`for i, j in combinations(range(N), 2): ...`) already want to iterate over, and empty-but-present entries mean no detector has to special-case "node i has zero pulses" with an `in` check.
- **Units pinned: frame indices, not seconds, not audio samples.** Same unit policy as `sustained_windows` — the helper stays config-free, and callers convert via `frames / control_rate_hz` when they need seconds. Rationale: (a) detector thresholds that matter for rhythm (flam window `10–40 ms`, phase-beating `Δf < 0.5 Hz`) are expressed in seconds, but those thresholds live with each detector, not with the primitive; (b) audio-sample indexing uses `audio_rate_hz` which has a different relationship to the state stream (22050 Hz vs 200 Hz) — collapsing either conversion into `pulse_times_of` would bake in a wrong assumption for the other use site. The primitive is the frame-index substrate; unit conversions are one multiplication away at the call site.
- **`DESIGN_V0.md §9.8` sketch intentionally diverged from.** The spec text lists `pulse_times_of(pulse_fired, t, i) → array of float seconds` — per-call single-node, seconds-valued, carrying the `t` array. Implemented as `pulse_times_of(pulse_fired) → tuple[np.ndarray, ...]` — whole-mask, frame-valued, no `t` needed. Reasoning: every detector that reads pulse times is inherently pairwise or all-vs-all over nodes (phase-beating pairs, flam pair-within-window, polyrhythmic multi-stream), so the per-node loop lives naturally at the helper boundary rather than inside every detector. And the `t` parameter is redundant once callers accept that frame indices are the unit — seconds conversion is `frames * dt` or `frames / control_rate_hz`, trivially one line at the consumer. The §9.8 text is a sketch, not a binding contract; deferring to the cleaner shape is cheaper than pre-committing and later renaming.
- **No companion helper landed.** Considered `inter_pulse_intervals(times)` and `pulse_times_seconds(pulse_fired, dt)`. Rejected both: intervals are a single `np.diff` call at the consumer and "the diff of a pulse-time sequence" is not a rhythm-ontology commitment worth naming; seconds conversion is a single multiplication. The brief explicitly asked for one clean primitive over early rhythm ontology, and every additional helper now pins a semantic the detector layer hasn't needed yet. The bar for a second helper is "two independent detectors both want it in the same shape"; today we have zero detectors that want it at all.
- **Dtype: `int64`, not `intp`.** `np.flatnonzero` returns `intp` (64-bit on this dev machine, 32-bit on 32-bit platforms). Forcing `int64` via `.astype(np.int64, copy=False)` makes the contract platform-independent and the test's `arr.dtype == np.int64` assertion stable across environments. The `copy=False` means no-op on 64-bit platforms (the common case).
- **Edge-case matrix exercised in `tests/test_derived.py`.** `T == 0` (every node present as empty `int64`), `N == 0` (empty tuple), all-False input (every node empty), all-True input on `(6, 2)` (every node `arange(6)`), node with zero pulses surrounded by nodes with pulses (present as empty `int64`, not `None` or omitted), int-castable non-bool input (`np.uint8`) accepted, 1D and 3D input rejected with `ValueError`. Tests are stdlib-only and run standalone without `pytest` just like `test_regimes.py` and `test_detectors.py`.
- **Smoke-checked on a real run.** On `regime_locked`'s 6 s / `T=1201` / `N=8` output: each node fires 13–14 times at the ~2.2 Hz near-consensus rate, all sequences strictly ascending, all `int64`. First pulse frames (60–70) correspond to the initial transient settling into lock — the detector layer will ignore the leading pulse before it's stable.
- **Not done in this pass.** No detectors (`phase_beating`, `flam`, `polyrhythmic` stay unimplemented — they need pairing logic, frequency estimates, window policies that this primitive doesn't prescribe). No intervals helper. No seconds conversion helper. No changes to `summary.json` schema, terminal oracle, or browser demo — the new primitive has no UI surface yet. No update to `DESIGN_V0.md §9.8`'s sketch signature; that text is acknowledged as the spec-at-time-of-writing and the working helper is documented in its own docstring plus this entry.
- **Next narrow step after this pass.** `detect_phase_beating(pulse_times, phase_vel, control_rate_hz)` is the honest next detector: (a) it's the simplest pulse-time-driven detector to define rigorously (pair up near-frequency nodes, build the pulse-arrival-offset series, look for a dominant low-frequency component); (b) it exercises `pulse_times_of` at the exact call shape it was designed for — pairwise iteration over `(out[i], out[j])`; (c) it produces the third detector in the matrix, making the two_cluster fixture fire *something* (it's currently silent on both detectors, a known coverage gap). `flam` and `polyrhythmic` can follow once phase-beating's pairing and frequency-filter helpers exist — they'll reuse the same machinery at different parameter settings.

### 2026-04-21 — Claude Opus 4.7 (doc pass 16 — first rhythm-oriented detector)
`detect_phase_beating` lands as the third detector and the first one built on `pulse_times_of`. Pulse-domain honest: it claims "the offset between two near-frequency pulse trains drifts linearly across the analysis window", not acoustic continuous-tone beating. Detector matrix grows from 2/13 to 3/13.

- **Detector shape: pairwise, one global window.** For each unordered pair `(i, j)` with `|Δω̄| ∈ [0.25, 1.5]` rad/s (the "close but not locked" band), sample `theta[:, j]` at `i`'s pulse frames inside `[warmup, T)`, unwrap the result, and measure two quantities: (a) total coverage `|seq[-1] − seq[0]|` — must be `>= 2π` rad, i.e. one full beat cycle actually completed inside the window; (b) `R²` of a linear fit — must be `>= 0.9`. A pair passes iff both hold. The detector fires iff at least one pair passes. `DetectionResult.windows` is a single analysis-window entry `[(warmup_frames, T)]` — this detector does not carve per-pair sub-windows; the evidence is a property of the whole window, not a transient event.
- **Velocity gate chosen to cleanly separate from the existing regimes.** Empirical survey across the three original fixtures (skipping the 1 s warmup, using the rest as the analysis window): `regime_locked` has every pair at `|Δω̄| ≈ 0` (all nodes tightly co-locked at 13.94 rad/s); `regime_two_cluster` has intra-cluster `|Δω̄| ≈ 0` and inter-cluster `|Δω̄| ≈ 5.58 rad/s` (clean bimodal fingerprint); `regime_drifting` has ten candidate pairs with `|Δω̄| ∈ [1.59, 5.96] rad/s`. Upper gate at 1.5 rad/s excludes the drifting minimum by `0.09` rad/s (about `7×` the `eta=0.03` noise-driven standard error over 8 s — a safe margin). Lower gate at 0.25 rad/s excludes the locked pairs by the width of the lower band. Both gates are round numbers; neither sits on a tuning knife-edge.
- **Drifting is silent, by design, even though it has structured pairwise drift.** `regime_drifting`'s candidate pair (1, 7) at `|Δω̄| = 1.59 rad/s` produces a textbook linear phase-difference drift with `R² = 1.000` and `~1.5` cycles of coverage across the run. Geometrically it IS beating. But "close in mean phase velocity" is the detector's load-bearing phrase, and 1.59 rad/s — a `12.8%` relative frequency spread — is on the outside of what this pass chooses to call "close". The alternative interpretation (fire on all near-linear phase drift regardless of magnitude) would make `phase_beating` and `drifting` partially redundant on the same fixture. Keeping the upper gate tight preserves a clean one-regime-one-detector picture at the cost of requiring a dedicated fixture for the positive case.
- **Two_cluster is silent, by design.** Intra-cluster pairs are locked (`|Δω̄| ≈ 0`, below the lower gate). Inter-cluster pairs have `|Δω̄| ≈ 5.58 rad/s`, beat period `~1.12 s` — fast enough to arguably count as structured drift but far above the upper gate. Either gate alone would silence two_cluster; together they form a comfortable moat around the 0.25–1.5 rad/s "close but not locked" band.
- **`configs/regime_phase_beating.yaml` added as the dedicated positive fixture.** Two nodes at 2.00 and 2.20 Hz (Δω ≈ 1.257 rad/s intrinsic; observed 1.223 rad/s after weak coupling pulls them slightly toward each other) sit close together in the unit square; six distractor nodes at 4.0, 4.5, 5.0, 5.5, 6.0, 7.0 Hz are clustered in the far corner (adjacent `|Δf|` gaps all `>= 0.5 Hz ≈ 3.14 rad/s`, far above the detector's upper gate, so no distractor pair enters the candidate band, and distractor-to-beating-pair Δf is `>= 1.8 Hz`, far above). `K0 = 0.30 rad/s` with `σ = 0.15` gives the beating pair an effective coupling of `~0.15 rad/s` — below the `|Δω|/2 ≈ 0.63 rad/s` two-oscillator Kuramoto lock threshold, so the pair drifts rather than locks. Duration 12 s covers ~2.4 full beat cycles at the observed `~5.14 s` beat period. `eta = 0` keeps the beating signature deterministic. Measured outcome: detector fires with confidence 0.098 (R² = 0.998 against a 0.9 floor → margin ~0.1), exactly one qualifying pair `(0, 1)`.
- **Pair metadata via a private helper, not a new `DetectionResult` field.** `_phase_beating_pair_evidence(theta, phase_vel, pulse_fired, control_rate_hz)` returns a list of dicts (`i, j, dw, slope, r2, coverage_rad, n_pulses_ref`) — one per qualifying pair. Tests consume it directly to assert that exactly the `(0, 1)` pair contributed to the beating fixture's fire. The detector itself wraps the helper and collapses the list into the uniform `DetectionResult` shape. Chosen over extending `DetectionResult` with an optional trailing field because (a) the brief was explicit about not broadly redesigning the detector result system, and (b) pair detail is debug-shaped, not downstream-consumer-shaped — a future `labels.json` writer doesn't want to serialize it.
- **Confidence formula: mean `(R² − 0.9)` across qualifying pairs.** Scale `[0, 0.1]` in practice (`R²` saturates near 1 for clean beating). Parallels `detect_phase_locked`'s `mean(r − R_LOCK)` — both are "how far inside the defining region was the evidence, averaged over what qualified". Alternative considered and rejected: confidence as the ratio of qualifying pairs to all near-frequency candidate pairs. Rejected because the numerator-over-denominator interpretation invites the question "what's a 50% confidence?" and the answer ("half the pairs in the velocity band were structured, half weren't") is a different phenomenon from the one this detector is trying to name.
- **Warmup skip = 1.0 s — big enough to outlast `regime_locked`'s transient and cheap everywhere else.** `regime_locked` takes about 0.25 s to reach r ≥ 0.9; the 1.0 s skip is four times that, conservative but not extravagant. For `regime_drifting`, `regime_two_cluster`, and `regime_phase_beating` the dynamics are stationary from the first sample, so the skip costs only pulse-count margin. Encoded as a seconds-valued constant (`PHASE_BEATING_WARMUP_S`) and converted to frames at call time, matching the existing `detect_phase_locked` / `detect_drifting` convention.
